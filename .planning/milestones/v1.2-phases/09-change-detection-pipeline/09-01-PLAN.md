<objective>
Create the automated change detection pipeline that monitors BCN.cl legal articles weekly,
detects text changes via SHA256 hash comparison, evaluates impact using DeepSeek AI,
and creates consolidated GitHub Issues when changes are found.

Purpose: Ensure calendarioescolar.cl feriado claims stay current with legal text changes.
When the Chilean congress modifies a holiday law, the team is automatically notified with
a diff, AI assessment, and recommended action.

Output:
- `scripts/check-bcn-changes.py` — standalone pipeline script
- `.github/workflows/check-bcn-changes.yml` — weekly cron + manual trigger Action
</objective>

import sys, os, importlib

bcn.fetch_norma_json(id_norma, timeout=30)
#   -> dict (BCN JSON response with 'estructura' and 'html' arrays)
#   Raises RuntimeError on HTTP error or URLError

bcn.extract_articles(norma_data)
#   -> list of {'numero': str, 'item_id': int, 'texto': str}

bcn.extract_text(html_string)
#   -> str (cleaned text, HTML stripped, entities unescaped)

bcn.compute_hash(text)
#   -> str (64-char SHA256 hex digest of UTF-8 encoded text)

bcn.get_feriado_claims(afirmaciones)
#   -> dict {source_id: [claim, ...]} — 4 keys (one per BCN law)

bcn.get_idnorma(afirmaciones, source_id)
#   -> str (idNorma number, with IDNORMA_CORRECTIONS applied)
#   Raises ValueError if not found

bcn.find_article_text(articles, articulo_numero)
#   -> str or None (article text matching ordinal/numeric name)

bcn.build_ai_client()
#   -> openai.OpenAI instance configured for DeepSeek (reads DEEPSEEK_API_KEY)

bcn.IDNORMA_CORRECTIONS
#   -> dict {'bcn-ley-20148': '257080'}

  <action>
  Create `scripts/check-bcn-changes.py` with the following structure. Use `var`-style Python (no type hints, no f-strings — match bcn-extractor.py style). All external HTTP calls via `urllib.request` (stdlib). Import openai lazily inside function.

  **1. Imports and bcn-extractor reuse (top of file):**
  ```python
  import sys, os, importlib
  sys.path.insert(0, os.path.dirname(os.path.abspath(__file__)))
  bcn = importlib.import_module('bcn-extractor')
  ```
  Plus: `import json, argparse, urllib.request, urllib.error` and `from datetime import datetime`.

  **2. `evaluate_impact(client, data_key, claim_text, texto_anterior, texto_nuevo)` function (CHNG-02):**
  - Builds a prompt asking DeepSeek to evaluate if the legal text change affects the claim.
  - Prompt includes: data_key, claim text, old article text, new article text.
  - Calls `client.chat.completions.create(model='deepseek-chat', max_tokens=16, messages=[...])`.
  - Parses response: if 'actualizar' in raw -> 'actualizar', if 'revision' in raw -> 'requiere_revision', else 'sin_impacto'.
  - Returns one of: `'sin_impacto'`, `'requiere_revision'`, `'actualizar'`.

  **3. `create_github_issue(title, body, labels)` function (CHNG-03):**
  - Reads `GH_TOKEN` from `os.environ.get('GH_TOKEN')`. If missing, raise RuntimeError.
  - Reads `GITHUB_REPOSITORY` from `os.environ.get('GITHUB_REPOSITORY')`. If missing, raise RuntimeError with message about running outside Actions.
  - POSTs to `https://api.github.com/repos/{repo}/issues` with headers:
    - `Authorization: Bearer {gh_token}`
    - `Accept: application/vnd.github+json`
    - `Content-Type: application/json`
    - `X-GitHub-Api-Version: 2022-11-28`
  - Payload: `{"title": title, "body": body, "labels": labels}`
  - Returns `html_url` from response.
  - On HTTPError, raises RuntimeError with status code and body.

  **4. `build_issue_body(changes, evaluations, claims_by_key)` function (CHNG-03):**
  - Builds consolidated markdown Issue body with 4 required components per change:
    1. Diff (texto anterior vs texto actual in code blocks)
    2. AI evaluation state (`sin_impacto|requiere_revision|actualizar`)
    3. Affected claims (data_key + claim text)
    4. Recommendation text based on state
  - Header: `## Cambios detectados en articulos BCN` + detection date.
  - Per change section: `### Claim afectado: \`{data_key}\`` with all 4 components.
  - Use `texto_antes` field from change dict (which is the stored texto_verbatim before overwrite) as "antes", NOT texto_anterior from JSON (per Pitfall 6 in RESEARCH.md).

  **5. `main()` function — orchestration:**

  a. **argparse**: `--dry-run` flag (skip Issue creation + DeepSeek calls, still fetch BCN to detect changes).

  b. **Load data files** (resolve paths relative to script via `os.path.dirname(os.path.abspath(__file__))`):
     - `data/afirmaciones.json` -> `afirmaciones`
     - `data/legal-articles.json` -> `legal_articles`

  c. **Build claims lookup**: `claims_by_key = {c['data_key']: c for c in afirmaciones.get('claims', [])}`

  d. **Iterate all entries in legal_articles** (skip `_meta`). For each `data_key`:
     - Skip if no `hash_sha256` (unidentified claim).
     - Get `id_norma` via `bcn.get_idnorma(afirmaciones, stored['ley_id'])`.
     - Group entries by `ley_id` to avoid fetching same law multiple times (important: entries like feriado_ano_nuevo, feriado_viernes_santo share bcn-ley-2977). Fetch each law ONCE.
     - Call `bcn.fetch_norma_json(id_norma)` — if RuntimeError: print to stderr, set `bcn_error = True`, BREAK the loop (do not continue to other laws). Per locked decision: BCN down = silent failure, no last_checked update.
     - Call `bcn.extract_articles(norma_data)` — if empty list, treat as BCN partial error, skip this law.
     - For each claim of this law: `bcn.find_article_text(articles, stored.get('articulo_numero'))` — if None, skip claim.
     - `new_hash = bcn.compute_hash(new_texto)`.
     - If `new_hash != stored_hash`: append to `changes` list with `data_key`, `texto_antes` (= stored `texto_verbatim`), `texto_despues` (= new_texto), `new_hash`.

  e. **If `bcn_error`**: print warning to stderr, exit 0 (silent failure). Do NOT update last_checked. Do NOT create Issue.

  f. **If changes detected and NOT `--dry-run`**:
     - Build DeepSeek client via `bcn.build_ai_client()`.
     - For each change: call `evaluate_impact(...)` to get state. Store in `evaluations` dict.
     - Build Issue body via `build_issue_body(changes, evaluations, claims_by_key)`.
     - Title: `'Cambio detectado en articulado BCN ({N} claim(s))'.format(N=len(changes))`.
     - Labels: `['bcn-change', 'legal-review']`.
     - Call `create_github_issue(title, body, labels)`. Print Issue URL.
     - Update legal_articles: for each change, set `texto_anterior = stored texto_verbatim`, `texto_verbatim = new texto`, `hash_sha256 = new_hash`.

  g. **If changes detected and `--dry-run`**: print changes summary to stdout (data_key, hash mismatch). Do NOT call DeepSeek or create Issue.

  h. **Update `last_checked`** for ALL entries (not just changed ones) to current UTC ISO timestamp. Also update `_meta.last_checked_at`.

  i. **Write back `legal-articles.json`** with `json.dump(..., ensure_ascii=False, indent=2)`.

  j. **Print summary**: `{N} changes detected, {M} claims checked. last_checked updated.`

  k. `sys.exit(0)`.

  **Important implementation details:**
  - Group claims by `ley_id` to fetch each BCN law only once (there are 4 laws, 15 claims — many claims share the same law).
  - The `texto_antes` in the Issue diff should be `stored_entry.get('texto_verbatim')` (the current stored text BEFORE overwrite), NOT `stored_entry.get('texto_anterior')` (which is from a previous run's change).
  - When BCN is unavailable for ANY law, stop processing ALL laws (break, not continue) — this ensures partial data does not create misleading Issues.
  - Exit code is always 0 (even on BCN error) — the Action should not fail on BCN unavailability.
  </action>

  **Workflow structure:**

  ```yaml
  name: Check BCN Legal Changes

  # Weekly check + manual trigger for testing
  on:
    schedule:
      - cron: '0 6 * * 1'    # Lunes 06:00 UTC (03:00 Chile)
    workflow_dispatch:         # Manual trigger from Actions UI

  jobs:
    check-changes:
      name: Detectar cambios en articulos BCN
      runs-on: ubuntu-latest
      permissions:
        contents: write        # Commit last_checked to legal-articles.json
        issues: write          # Create GitHub Issues when changes detected

      steps:
        - name: Checkout
          uses: actions/checkout@v4
          with:
            token: ${{ secrets.GITHUB_TOKEN }}

        - name: Setup Python
          uses: actions/setup-python@v5
          with:
            python-version: '3.x'

        - name: Install dependencies
          run: pip install openai

        - name: Check BCN for changes
          env:
            DEEPSEEK_API_KEY: ${{ secrets.DEEPSEEK_API_KEY }}
            GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          run: python scripts/check-bcn-changes.py

        - name: Commit updated last_checked
          if: always()
          run: |
            git config user.name "calendarioescolar-bot"
            git config user.email "bot@calendarioescolar.cl"
            git add data/legal-articles.json
            git diff --staged --quiet || git commit -m "chore: update legal-articles last_checked [skip ci]"
            git push origin ${{ github.ref_name }}
  ```

  **Key details:**
  - `GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}` maps the auto-token as `GH_TOKEN` env var, which the Python script reads via `os.environ.get('GH_TOKEN')`. This avoids needing a separate PAT secret.
  - `permissions: issues: write` is REQUIRED for creating Issues with GITHUB_TOKEN.
  - `permissions: contents: write` is required for committing last_checked updates.
  - `[skip ci]` in commit message prevents deploy.yml from triggering on the last_checked update commit.
  - `if: always()` on the commit step ensures last_checked is committed even if the script reports changes (which is not a failure — script always exits 0).
  - `pip install openai` installs the openai SDK needed for DeepSeek API calls.
  - `actions/setup-python@v5` with `python-version: '3.x'` matches latest stable Python.
  - Bot name `calendarioescolar-bot` and email `bot@calendarioescolar.cl` match the established pattern from sync-deploy.yml.
  </action>
