    Rewrite sync-from-sheet.js to read from a single "Datos" tab instead of "Regiones" + "Config" tabs. Keep the same coding style: var, require, no ES modules, IIFE-compatible, native https only.

    PRESERVE these existing elements:
    - The fetchSheet() helper function (works fine, just called with different tab name)
    - The cell() helper function
    - The writeIfChanged() helper function
    - The GOOGLE_API_KEY auth pattern
    - The config.json reading pattern
    - The error handling and timeout patterns

    CHANGE the configuration section:
    - Remove REGIONS_TAB and CONFIG_TAB variables
    - Add: var DATOS_TAB = (config.sheet && config.sheet.datosTab) || 'Datos';
    - Update range to 'A1:L100' (12 columns, up to ~73 rows: 1 header + 50 claims + 16 regions + 7 config)

    REPLACE parseRegionsTab() and parseConfigTab() with a single parseDatosTab(rows) function that:
    1. Skips row 0 (headers)
    2. For each row, reads col A (seccion) to route to the correct section
    3. CLAIMS section (seccion === 'CLAIMS'):
       - Build a claim object: { id: col B, pregunta: col C, respuesta: col D, fuente_url: col E, source_reference: col F (fuente_referencia), extracto_verbatim: col G, hash_sha256: col I (hash_verbatim), last_checked: col J, status: col K }
       - Collect into claims array
    4. REGION section (seccion === 'REGION'):
       - col D (respuesta) contains JSON.stringify of region object — JSON.parse it
       - Collect into pages array
    5. CONFIG section (seccion === 'CONFIG'):
       - col B (id) is the key name, col D (respuesta) is the value
       - For 'year': parseInt the value
       - For 'feriados' and 'feriadosCompletos': JSON.parse the value (it's a stringified array)
       - For 'schoolStart', 'winterStart', 'winterEnd', 'schoolEnd': string value
       - Build calConfig object with same shape as current calendar-config.json

    REPLACE the main() flow:
    - Fetch only ONE tab (DATOS_TAB) instead of two parallel fetches
    - Call parseDatosTab(rows) to get { claims, pages, calConfig }
    - Build claims.json output:
      - Read existing data/claims.json to get _meta and sources sections (preserve them)
      - Replace claims array with Sheet data, merging Sheet fields into existing claim objects:
        - For each Sheet claim, find matching claim in existing claims.json by id
        - If found: update respuesta, pregunta, fuente_url, source_reference (from fuente_referencia), extracto_verbatim, hash_sha256 (from hash_verbatim), last_checked, status from Sheet
        - If not found: create minimal claim object from Sheet data
        - Preserve all other fields from existing claim (tags, data_key, data_path, source_id, displayed_value, fuente_tipo, articulo_numero, inciso, texto_anterior, verification_method, verification_tier, expected_check, last_verified, last_verified_by)
      - Write with writeIfChanged
    - Write pages.json with writeIfChanged (same as before)
    - Write calendar-config.json with writeIfChanged (same as before)

    VALIDATION in parseDatosTab:
    - Throw if pages.length !== 16 ("Se esperan 16 regiones en la pestana Datos")
    - Throw if claims.length === 0 ("No se encontraron claims en la pestana Datos")
    - Throw if any required calConfig field missing (year, schoolStart, winterStart, winterEnd, schoolEnd)
    - Warn (console.warn) if a REGION row's respuesta fails JSON.parse

    UPDATE console output:
    - Print "Tab datos: DATOS_TAB" instead of separate tab names
    - Print summary: "Leidos N claims + M regiones + config (year: Y, feriados: F)"

    Add --dry-run flag support: if process.argv contains '--dry-run', print parsed data summary but skip writeIfChanged calls.

    Update the file header comment to document the new Datos tab structure (12 columns, 3 sections).
